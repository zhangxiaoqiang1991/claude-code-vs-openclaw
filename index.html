核心差异：两种设计哲学

Claude Code 和 OpenClaw 看起来越来越像（都有 memory、skill、定时任务），但出发点完全不同。不是谁抄谁，而是合理的 Agent 系统该有的东西都在那，但底层架构哲学差异巨大。

核心一句话：Claude Code 是为"深度推理协作"设计的终端编码工具（工匠的锤子），OpenClaw 是为"7×24 自动化编排"设计的个人 Agent 网关（数字员工）。

维度Claude CodeOpenClaw
设计原点 "让 Claude 能操作终端和文件系统，深度完成编码任务" "I wanted to tell my computer to do things, and it would just do them." — Peter Steinberger
部署形态 CLI 终端，session 生命周期 后台守护进程（gateway daemon）
运行方式 你主动打开 → 对话 → 关闭 7×24 在线，随时被动响应
交互通道 终端 stdio 20+ IM（WhatsApp/Telegram/Slack/Discord/…）
模型绑定 Anthropic Claude（深度优化） 20+ 供应商，可自由切换
编码能力 原生深度推理，SWE-bench ~80.9% 通过 bridge 调用 Claude Code/Codex/Cursor
上下文引擎 5 层自动化压缩管线 + auto-inject 可插拔（BM25+向量/Lossless-Claw DAG）
记忆策略 Pull 模式：索引在 context，内容按需读取 Push 模式：启动时全量加载
自修改能力 不可修改核心源码（闭源） 可以自读源码并修改（开源 TypeScript）
扩展方式 Skill + MCP + Hook + memory Skill（SKILL.md）+ 插件 + ClawHub
目标用户 开发者、内容创作者 极客、自动化发烧友、团队

OpenClaw 由奥地利开发者 Peter Steinberger 于 2025 年 11 月创建，已成为 GitHub 历史上增长最快的开源项目之一（36 万+ Star）。以下是近 30-60 天的重要更新：

  • 直进会议、实时语音 loop、导出记录

  • 统一 SQLite 任务账本

🛡 安全加固

  • 语义化工具审批、fail-closed 安装
  • 内联 eval 检测

OpenClaw 的 24/7 守护进程

不是"一直在聊天"，而是"一直在线等你需要它"。核心应用场景：

场景
异步任务编排
在 WhatsApp 说"帮我 review 那个 PR"，它在后台启动 Claude Code bridge 做 review，做完推送结果。不需要开终端、不需要等。
场景
跨系统自动化
查日历 → 订会议室 → 发确认邮件 → 加到 Trello，一个自然语言指令串起多个系统。
场景
长时间后台任务
跑数据分析、爬资料、做研究——不用守着终端，做完通知。
场景
IM 个人助理
Telegram / WhatsApp / Discord 里直接对话，不用切窗口。非技术用户也能用。

本质区别：Claude Code 是你主动找它干活；OpenClaw 是它一直在那等你找它。前者是深度协作工具，后者是 24 小时待命的数字员工。

自进化能力对比

这是两个系统最本质的区别之一。OpenClaw 能修改自己的源码，Claude Code 主要在知识层面进化。

OpenClaw 的自修改机制

层级机制例子
系统自感知 Agent 知道自己的源码位置、架构、配置文件在哪 —
自然语言 → 代码 用户说需求，Agent 自己读源码、写补丁 "帮我加个 OCR 功能" → Agent 读源码插代码
文件即记忆皮层 SOUL.md / AGENTS.md / MEMORY.md 均可自改写 犯错后自动记录到 AGENTS.md
自生成 skill AutoSkill：重复模式自动提取为 SKILL.md Agent 自己写新 skill 扩展
涌现能力 未预设的功能，Agent 自己组合工具完成 摩洛哥语音消息：自己调 ffmpeg + Whisper API

"None of this did I code. The mad lad figured it out on its own."

— Peter Steinberger, 关于 Agent 自己处理语音消息

Claude Code 的进化方式

进化维度具体做法状态
记忆积累 Auto memory：对话中提炼原则写入 MEMORY.md ✅ 已实现
人格配置 CLAUDE.md 定义行为、价值观、交互风格 ✅ 已实现
技能扩展 Skill 系统：用户编写 SKILL.md 固化工作流 ✅ 已实现
错误反馈闭环 用户纠错 → 记入 memory → 不犯第二次 ✅ 已实现

修改核心源码 修改 Claude Code 本身的能力 ❌ 不可（闭源）

关键认知：OpenClaw 的进化是"agent 自己改自己脑子"（无监督学习），Claude Code 的进化是"你指方向，我优化路径"（人在回路）。对于创作者场景，后者更可控、更稳定。

记忆系统设计哲学：Push vs Pull

这是解释为什么 OpenClaw 耗 token 而 Claude Code 不耗的核心原因：

维度OpenClaw（Push）Claude Code（Pull）
启动时 全量加载 SOUL + AGENTS + USER + MEMORY 仅加载 CLAUDE.md + MEMORY.md 索引（~30 行）
对话中 所有记忆一直在 context 里 索引在 context，具体内容按需读取
上下文开销 2-5 倍系统 prompt token 很轻量，~1K tokens
应对不确定 不知道用户会问什么，所以全部装上膛 知道当前 session 做什么，没必要全加载
优点 任何问题都能答，无需额外读取 轻量、省 token、专注
缺点 贵，context 容易被噪音稀释 需要 Agent 有判断力，知道何时该读什么

两种设计哲学：OpenClaw 相信"给模型看更多，让它自己判断"，Claude Code 相信"让 Agent 自己判断需要什么"。前者是通用性优先，后者是效率优先。

MEMORY.md 每条配一句话描述，让索引极轻但检索极准。这是对抗记忆臃肿的关键设计。

记忆写入原则：只记原则性的、可复用的方法论和决策框架；不记日常项目、一次性任务和对话细节。每条记忆都应该是将来可复用的信号，不是对话的存档。

7

Token 消耗分析

OpenClaw 的 token 消耗远高于 Claude Code，原因如下：

  1. 全量加载策略 — SOUL.md（~20K chars）+ AGENTS.md + USER.md + MEMORY.md 全部塞入 system prompt
  2. Gateway 层开销 — IM 协议解析、信道加密、路由转发，每层都加 token
  3. 模型无优化 — 通用 gateway 无法像 Claude Code 那样为单一模型做深度上下文压缩
  4. 24/7 常驻 — 即使 idle 也在消耗资源（心跳检测、状态同步）

一个直观对比

同样的对话量和记忆量，每次请求的 system prompt token 消耗：

OpenClaw:   SOUL.md(~5K) + AGENTS.md(~3K) + USER.md(~1K) + MEMORY.md(~5K) = ~14K tokens
Claude Code: CLAUDE.md(~1K) + MEMORY.md 索引(~0.5K) = ~1.5K tokens

差异：约 5-10 倍系统 prompt token

对你来说：每天 8-10 小时高强度使用 Claude Code，即使这样成本也远低于 OpenClaw 跑一个下午。你自己说的——"在 OpenClaw 上面，再多的钱都扛不住"。

SOUL.md vs CLAUDE.md

两个系统都有"人格配置文件"的概念，结构和用途高度对应：

概念OpenClawClaude Code
人格配置 SOUL.md —— 性格、语气、价值观 CLAUDE.md —— 灵魂设定 + 身份 + 工作偏好
操作规程 AGENTS.md —— 行为规则、经验教训 CLAUDE.md（同文件） + Memory
长期记忆 MEMORY.md Auto Memory（MEMORY.md）
能力模块 SKILL.md Skill（SKILL.md）
用户画像 USER.md CLAUDE.md（身份部分） + Auto Memory
自改写 ✅ Agent 可以改写自己的 SOUL.md ⚠️ 需用户确认后改写

OpenClaw 的配置层次

SOUL.md     — 怎么说话（开车风格）
USER.md     — 为谁服务（目的地）
AGENTS.md   — 怎么做事（导航路线）
MEMORY.md   — 学到了什么（经验积累）
SKILL.md    — 会做什么（工具箱）

Claude Code 的配置层次

CLAUDE.md   — 灵魂设定 + 人格 + 身份 + 工作偏好（一站式）
MEMORY.md   — 原则性长期记忆（按需加载）
SKILL.md    — 能力模块
Hooks       — 自动化行为触发
MCP         — 外部工具协议

一个值得注意的区别：OpenClaw 是分散式多文件结构（SOUL/AGENTS/USER/MEMORY各自独立），Claude Code 是集中式（CLAUDE.md 接管了大半）。集中式的好处是维护简单、不易遗漏；分散式的好处是职责清晰、可以分别自改写。

9

总结：什么场景用什么

✅ 选 Claude Code

  • 你是内容创作者 / 增长运营
  • 你需要深度推理和高质量输出
  • 稳定 > 灵活，可控 > 无限可能
  • 你每天用 8-10 小时，成本敏感
  • 你不想要"半夜自己改代码"的 Agent
  • 你在做需要深度思考的创作或编码

✅ 选 OpenClaw

  • 你需要 7×24 待命的数字员工
  • 你的用户/团队在多个 IM 平台上
  • 你要做一个 Agent 自进化的实验项目
  • 你需要跨系统编排（日历+邮件+Trello+…）
  • 你对 token 成本不敏感
  • 你有技术能力维护和排查问题

强哥的选择：Claude Code + 自建 skill 体系 + auto memory + 定时任务 + 飞书 MCP。用 Claude Code 做内核，自己搭一套轻量 OpenClaw-like 系统。稳定、省 token、不被锁定。

"一个工具好不好，不看它有多少功能，看你愿不愿意整天待在它里面。"
